## 서론

대규모 언어모델(LLM)은 자연어 생성 능력이 비약적으로 향상되면서 다양한 분야에 적용되고 있다. 그러나 생성 결과에 포함되는 사실 오류, 불확실한 추론, 출처 미기재의 세부사항 조작 등 '환각(hallucination)' 현상은 모델 신뢰성의 근본적 문제로 대두되었다. 본고는 환각 문제를 "진단 가능하고 재현 가능한" 방식으로 다루기 위해 분류학(taxonomy)을 제안하며, 연구자 및 실무자가 적용할 수 있는 실험 프로토콜과 운영화 체크리스트를 제시한다. 목표는 환각의 원인을 명확히 분리(prompt vs. model)하고, 상황별 완화 전략을 체계적으로 권고하는 것이다.

## 환각의 정의와 연구적 쟁점

환각은 표면적으로는 유창하지만 사실적으로 부정확하거나 근거가 없는 정보를 생성하는 현상으로 정의한다(예: 잘못된 연도, 존재하지 않는 인용, 맥락에 맞지 않는 세부사항). 주요 연구 질문은 다음과 같다: (1) 환각의 주요 원인은 무엇인가(프롬프트·데이터·모델 아키텍처·추론 샘플링 등)? (2) 환각의 유형을 실험적으로 분류할 수 있는가? (3) 운영 환경에서 환각을 어떻게 탐지·완화할 것인가?

이러한 질문에 답하기 위해 본고는 환각을 발생 원인(프롬프트-유발 vs 모델-내재)과 표현 형태(사실 오류, 논리적 불일치, 맥락 외 창작)라는 두 축으로 분류한다. 이후 섹션에서 실험적 지표(Prompt Sensitivity, Model Variability 등)와 검증 방법을 제시한다.

## 분류학: 원인 축과 표현 축

### 원인 축: Prompt-induced vs Model-intrinsic

프롬프트-유발 환각은 사용자의 요청이 모호하거나 유도적일 때 발생한다. 예컨대 "~라고 들었는데 사실인가?"와 같은 질문은 모델을 추측으로 몰아넣는다. 모델-내재 환각은 훈련 데이터의 불완전성, 레이블 오류, 또는 모델의 내부적 일반화 오류에서 기인한다. 실무적으로 두 원인을 구분하면 대응 전략(프롬프트 개선 vs 모델 재학습)이 달라진다.

### 표현 축: Factual / Logical / Extrinsic / Detail fabrication

- 사실적 오류(Factual): 실제 사실과 불일치하는 정보(날짜·이름·통계 등).
- 논리적 불일치(Logical): 추론 과정에서 내부 모순 발생(수학적 오류, 전제 불일치).
- 맥락 외 창작(Extrinsic fabrication): 주어진 문맥에서 벗어난 부가 정보 생성.
- 세부사항 날조(Detail fabrication): 근거 없는 인용·출처·수치 생성.

이 분류는 평가 지표와 자동 탐지 규칙(예: 출처 매칭 실패, NLI 기반 불일치 검사)을 설계하는 기초가 된다.

## 진단 지표와 실험 프로토콜

### Prompt Sensitivity(PS)

PS는 동일 모델에 여러 프롬프트 변형(제로샷, 퓨샷, CoT, 명시형 등)을 적용할 때 관찰되는 환각률의 변화량이다. PS가 높으면 프롬프트 개선으로 환각을 줄일 수 있음을 시사한다.

### Model Variability(MV)

MV는 동일 프롬프트에 대해 여러 모델을 비교했을 때 환각률의 분산이다. MV가 높으면 모델 구조·훈련 데이터의 영향이 크다는 의미다.

### Joint Attribution Score(JAS)

JAS는 프롬프트와 모델의 상호작용을 정량화한다. 특정 프롬프트-모델 조합에서 환각이 유독 높게 발생하면 JAS가 상승한다.

### 실험 프로토콜(재현성 원칙)

1. 데이터: TruthfulQA, QAFactEval, HallucinationEval의 하위셋(도메인별 균형 샘플).
2. 프롬프트 변형: Zero-shot / Few-shot(2–3 예시) / Instruction / CoT / Vague(의도적 모호성).
3. 모델군: GPT-4 계열, LLaMA 계열, 커뮤니티 모델(예: Mistral, Llama-derivatives).
4. 추론 파라미터: temperature, top-p, max tokens, sampling seeds를 고정·기록.
5. 평가: 자동 지표(QAFactEval, NLI 기반 검증, 출처 매칭) + 사람 평가(중요 샘플에 대해 3인 이상 주관성 투표).

이 프로토콜을 통해 PS·MV·JAS를 계산하고, 각 사례를 Prompt-dominant, Model-dominant, Mixed-origin, Noise로 분류한다.

## 완화 전략: 실무 가이드

### 프롬프트 단계

- 역할·출력형식·근거 요구를 명시하여 모호성 제거.
- CoT(Chain-of-Thought)를 통해 중간 추론을 유도하여 논리적 오류를 드러내기.
- 응답의 출처 표기(가능하면 문서 ID/URL) 규칙을 적용.

### 시스템 파이프라인

- RAG: 근거 문서를 검색하여 생성 시점에 인젝션(injection)함으로써 사실성 향상.
- 검증자(verification model): 생성 결과를 사실성 분류기로 평가. 불확실 시 인간 검토로 라우팅.
- Self-consistency: 다중 생성 샘플의 합의를 통해 신뢰도 점수 산출.

### 모델 수준

- Grounded fine-tuning: 신뢰 가능한 지식 소스 기반의 미세조정.
- RLHF: 사실성·정확성 보상을 포함한 보상 함수를 설계.
- Contrastive decoding: Hallucination-prone 토큰을 억제하는 디코딩 전략 적용.

## 운영화 체크리스트

1. Prompt Clarity Checklist(역할, 출력 포맷, 근거 정책, 시한 등) 도입.
2. 인덱스 신선도(Freshness)·출처 신뢰도(Trust) 지표 정의 및 모니터링.
3. 로그 정책: 입력/출력 로그의 민감도 분류·마스킹·접근 통제.
4. 버전 관리: 모델/프롬프트/인덱스 버전 태깅 및 회귀 테스트 자동화.
5. SLA 기반의 인간 개입 정책: 민감 쿼리·임계값 초과 시 인간 검토.

## 향후 연구 방향

- 멀티모달 환각: 비전-언어 모델에서의 사실성 검증 방법.
- 실시간 검증 파이프라인의 지연-정확성 트레이드오프 최적화.
- 표준화된 환각 벤치마크 및 평가 메트릭 개발.

## 결론

LLM 환각은 복합적 원인에서 발생하며, 프롬프트 개선과 모델 개선을 병행해야 한다. 본 논의는 환각의 분류학과 진단 지표를 제안하고, 운영 환경에서 적용 가능한 완화 전략과 체크리스트를 제공한다. 연구와 운영의 밀착이 환각 문제 해결의 핵심이다.
